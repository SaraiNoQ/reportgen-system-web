# 开发流程

## 1. 需求确认

开发前先阅读：

- `docs/需求规格说明书.md`
- `DESIGN.md`
- `docs/前端需求说明.md`
- `docs/前端开发规范SPEC.md`

当 Figma 初版原型与需求规格不一致时，以需求规格为准。

## 2. 页面开发

1. 明确页面所属模块和路由。
2. 在 `src/lib/types/domain.ts` 补齐前端领域类型，并与后端 Pydantic schema 保持字段一致。
3. 在 `src/lib/services/api.ts` 增加 adapter 方法。
4. 若页面需要只读 fallback，再在 `src/lib/mock/data.ts` 增加演示种子；写操作不得只落在 mock 数据。
5. 在 `src/app` 增加页面文件。
6. 如果有复杂交互，在 `src/components/pages` 增加 client component。
7. 优先复用 `src/components/ui` 和 `src/components/layout` 中的组件。

## 3. UI 调整

- 先改共享组件 token，再改页面局部样式。
- 保持字体大小稳定，优先通过 padding、gap、列宽、控件高度和圆角调整密度。
- 工作台页面不做营销型 hero，不使用大面积装饰渐变。
- 表格与编辑区以清晰、紧凑、可扫描为主。

## 4. 验证

每轮较大修改后执行：

```bash
pnpm lint
pnpm typecheck
pnpm build
```

前端视觉改动还应启动：

```bash
pnpm dev
```

并检查以下页面：

- `/login`
- `/forgot-password`
- `/account`
- `/records`
- `/rules`
- `/reports`
- `/projects`
- `/system/users`
- `/system/logs`

## 5. Core API 联调

当前页面已经通过 `src/lib/services/api.ts` 接入 Core API（含 gen-report 工作流端点）。新增或调整功能时按以下顺序处理：

1. 在后端 `app/schemas/domain.py` 定义请求和响应。
2. 在后端 `app/services/mock_store.py` 和 `app/services/postgres_store.py` 增加或调整双数据层读写逻辑。
3. 如涉及 schema 变更，在 `app/models` 和 `alembic/` 新增 ORM 模型和迁移。
4. 在后端 `app/api/v1` 暴露路由，并补充 `backend/tests`。
5. 在 `backend/docs/Core API 接口说明.md` 同步接口文档。
6. 在前端 `src/lib/services/api.ts` 增加 adapter 方法（gen-report 工作流相关方法放在 `genReportApi` 命名空间下）。
7. 页面组件只调用 adapter，不直接访问 `fetch` 或 `src/lib/mock/data.ts`。
8. 执行 `uv run ruff check .`、`uv run pytest`、`pnpm lint`、`pnpm typecheck`、`pnpm build`。

页面组件不应直接依赖具体 HTTP client，应继续通过 adapter 调用。
