# 技术栈说明

## 总体选择

后端采用 Python 技术栈，原因是大模型端已经使用 Python，实现、调试和复用成本最低。当前推荐形态是模块化 Core API 加独立 AI Worker，而不是一开始拆成完整微服务。

```text
FastAPI Core API
  -> PostgreSQL
  -> Redis/Celery
  -> MinIO/S3
  -> Python AI Worker
```

## Core API

- FastAPI：HTTP API、OpenAPI 文档、依赖注入。
- Pydantic v2：请求和响应结构校验。
- pydantic-settings：环境变量和 `.env` 配置。
- SQLAlchemy 2.x：数据库 ORM。
- Alembic：数据库迁移。
- PostgreSQL：核心业务数据库。
- python-jose：JWT 签发和验证。
- passlib[bcrypt]：密码哈希。
- structlog：结构化日志。

## 数据层

后端支持 mock JSON 和 PostgreSQL 双数据层，通过 `STORAGE_BACKEND` 环境变量切换：

- **mock 模式**（默认）：使用仓库根目录 `data/` 下的 JSON 表保存数据。后端启动时如果 `data/` 不存在，会用内置种子数据创建表文件；写接口会同步回写 JSON 文件。
- **postgres 模式**：通过 `postgres_store.py` 和 `repositories/` 访问 PostgreSQL，schema 变更由 Alembic 迁移管理。

两种模式下 API schema 和前端 adapter 签名保持一致。pytest 通过 `INSPECTION_DATA_DIR` 使用临时目录，避免污染开发数据。

## gen-report 工作流

gen-report 工作流是 Core API 内置的报告生成管线，包含四个阶段：validate → prepare → extract → generate。

- `manifest_builder.py`：根据项目规则模板和源文件生成 manifest.yaml 和工作空间目录。
- `gen_report_service.py`：编排工作流运行、审批、字段设置和报告生成，维护 run 级状态和工作空间。
- 外部 Python 包 `gen-report` 提供 validate/prepare/extract/generate 四阶段的实际执行能力。

## 异步任务

- Celery：后台任务调度、重试、失败记录。
- Redis：开发和中小规模部署阶段的 Broker/Backend。

解析任务、Word 导出、批量导入用户等耗时操作都应通过任务队列执行，HTTP 请求只负责创建任务和返回任务 ID。

## 文件存储

- MinIO：本地或私有化部署对象存储。
- S3 协议：便于后期替换到云厂商对象存储。

数据库只保存文件元数据、对象存储 key、摘要、大小、类型和业务状态，不保存原始文件二进制。

## AI Worker

AI Worker 使用 Python，优先封装已有命令行解析程序：

- `subprocess` 或内部 Python API 调用现有解析 pipeline。
- 输入来自 Core API 创建的解析任务。
- 输出结构化 JSON、置信度、来源位置和错误摘要。

Worker 可以和 Core API 同仓库开发，也可以后期独立部署。边界通过任务 payload 和结果 schema 保持稳定。

## 开发工具

- uv：依赖安装和虚拟环境管理。
- pytest：测试。
- httpx：API 测试客户端。
- ruff：格式和静态检查。

## 环境变量

核心配置见 `.env.example`：

- `DATABASE_URL`
- `REDIS_URL`
- `OBJECT_STORAGE_*`
- `SECRET_KEY`
- `AI_TASK_QUEUE`

生产环境不得使用 `.env.example` 中的默认密钥。
