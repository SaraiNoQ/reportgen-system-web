# 开发流程

## 1. 需求确认

开发前先确认：

- 需求所属模块：认证、用户、项目、文件、解析任务、规则、报告、日志。
- 是否涉及 AI Worker。
- 当前是否需要新增或调整根目录 `data/*.json` 表；后续迁移真实数据库时再评估 Alembic 迁移。
- 是否需要对象存储或任务队列。
- 前端页面是否已有 adapter 调用约定。

如需求会影响大模型输入输出 schema，必须先和 AI 侧确认字段契约。

## 2. 接口设计

1. 在 `docs/后端需求说明.md` 或具体接口文档中补充业务说明。
2. 定义请求、响应和错误码。
3. 明确权限要求。
4. 明确是否同步返回结果，还是创建异步任务。
5. 明确日志和审计要求。

## 3. 数据建模

当前开发期：

1. 在 `app/schemas/domain.py` 定义 Pydantic 请求和响应模型。
2. 在 `app/services/mock_store.py` 增加或调整内存种子数据、`data/*.json` 表名和读写方法。
3. 对需要追溯的业务数据保留历史表或事件表，例如 `operation_logs.json`、`report_versions.json`、`rule_template_versions.json`。
4. pytest 必须继续使用 `INSPECTION_DATA_DIR` 临时目录，避免污染根目录数据。

后续接入 PostgreSQL 时：

1. 在 `app/models` 新增或调整 ORM 模型。
2. 在 Alembic 中生成迁移。
3. 检查索引、唯一约束和外键。
4. 保持 API schema 和前端 adapter 签名稳定。

## 4. 代码实现

推荐顺序：

1. `schemas`：定义输入输出。
2. `services/mock_store.py`：定义 JSON 表读写和业务状态变更。
3. `api`：暴露 HTTP 路由。
4. `tests`：补充接口和服务测试。
5. `docs/Core API 接口说明.md`：同步接口说明、请求响应和数据表影响。
6. 前端 `src/lib/services/api.ts`：同步 adapter。

后续数据库化时，再引入 `models`、`repositories` 和 Alembic 迁移。

## 5. 解析任务接入流程

真实 AI 能力接入时推荐顺序：

1. Core API 创建 `parse_job`。
2. Core API 校验项目、文件、模板和权限。
3. Core API 将任务 payload 投递到队列。
4. AI Worker 消费任务并写入阶段进度。
5. AI Worker 产出结构化结果。
6. Core API 保存结果并更新项目/文件/报告状态。
7. 前端通过轮询或 SSE 查询任务进度。

## 6. 本地验证

每轮后端修改至少执行：

```bash
uv run ruff check .
uv run pytest
```

启动服务：

```bash
uv run uvicorn app.main:app --reload --host 127.0.0.1 --port 8000
```

健康检查：

```bash
curl http://127.0.0.1:8000/api/v1/health
```

## 7. 推荐开发顺序

1. 健康检查和配置加载。
2. 登录鉴权。
3. 用户和角色管理。
4. 项目列表和项目详情。
5. 文件元数据、解析事件和字段结果。
6. 解析状态更新、失败重试和人工补录。
7. 规则模板、字段编辑和版本管理。
8. 报告生成、章节编辑、导出、预览和回退。
9. 操作日志、消息、用户偏好和项目恢复。
10. 对象存储、真实解析任务和 AI Worker。
